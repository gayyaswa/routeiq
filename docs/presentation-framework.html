RouteIQ — Presentation Framework

1. Use Case

RouteIQ answers scenic travel queries in natural language: "Plan a day trip in SF focused on hiking and historic sites" → returns a timed itinerary with real stops, photos, ratings, and a narrative. The hard problem is spatial + semantic alignment: the right POIs aren't just semantically close to the query — they have to be geographically reachable and worth the detour.

2. Where RAG Fits

Three distinct retrieval layers working together:

Layer What it retrieves Why RAG (not just a DB lookup)
Graph RAG OSM road network → spatially joined POIs along actual route Spatial proximity alone doesn't capture "scenic" — need detour scoring
Vector RAG ChromaDB → Wikipedia descriptions for each candidate POI POI names are poor search keys; descriptions enable semantic reranking
Knowledge RAG KnowledgeRAG 3-stage pipeline: (1) ChromaDB vector search by preferences → (2) KG graph filter (on-route check) + augment (city, region, nearby POIs via LOCATED_IN / NEAR_POI edges) → (3) enriched context string for LLM narrative Vector search alone gives semantically relevant POIs but no structural context; KG enrichment adds "what's nearby", "which city", "which region" — grounding the narrative in real graph relationships
Live RAG Tavily web search → real-time activity context ("best hiking in SF") OSM tags miss activity quality signal entirely

Key insight: each layer handles a different failure mode. Graph RAG anchors stops to the real route spatially. Knowledge RAG combines vector similarity with structured entity relationships. Live RAG fills quality signal OSM can't provide. Together they cover spatial, semantic, structural, and freshness dimensions of retrieval.

3. Tools the Agent Calls

7 tools total:

Tool What it does
find_city_poisKnowledgeGraph lookup — returns up to 100 POIs for a city by OSM category; instant, no network call
select_pois_for_dayActivity-matched + scenic fill merge — two-track selection when the user specifies activities (hiking, kids, etc.)
rate_poisTripAdvisor or LLM-synthetic rating per POI; fetches Wikipedia description + composite scoring (rating × review count × Wikipedia weight)
query_poi_contextIndexes rated POIs into ChromaDB, then retrieves KG-enriched context (semantic match score + city/region + NEAR_POI edges) for grounded narrative generation
enrich_poi_detailsFetches Wikipedia description + thumbnail image URL for a specific named POI — used during refinement or when rate_pois cache misses
estimate_visit_durationReturns typical visit time in minutes by OSM category + subtype (e.g. museum → 90 min, viewpoint → 30 min)
search_poi_by_nameNominatim geocoding — resolves a user-named landmark ("add Lombard Street") to a POI dict during itinerary refinement

4. Autonomy vs. Workflow

Mode Where it applies What's fixed / what's free
Workflow LangGraph pipeline nodes Shape is fixed: parse → graph → rag → narrate always runs in order
Autonomy ReAct tool loop inside the graph node Agent decides which tools to call, how many times, and retries if recall is low
Human-in-the-loop interrupt() before narrate User sees draft itinerary, can refine ("add food stops"), agent replans

Honest framing — medium autonomy. The pipeline shape is locked (workflow), but tool selection within nodes is agent-driven. This is the right trade-off: full autonomy would make evaluation unreliable.

5. How We Evaluate It

15 queries × 5 configs comparing OSM vs. Tavily classifiers and LLM-synthetic vs. TripAdvisor ratings:

Config Pass Rate Tool Routing Activity Recall Avg Time
Run 1 — OSM + Synthetic13 / 1515 / 1583%66.0s
Run 2 — Tavily + Synthetic15 / 1515 / 15100%70.3s
Run 3 — Tavily + Enrich14 / 1515 / 1590%49.6s
Run 4 — OSM + TripAdvisor13 / 1515 / 1583%41.9s
Run 5 — Tavily + TripAdvisor ✓15 / 1515 / 1597%43.8s

Metrics tracked (5 total):

  • Tool routing accuracy — did the agent call the right POI discovery tool first? (select_pois_for_day when activities requested, find_city_pois otherwise) — 15/15 across all configs
  • Activity recall — fraction of requested activities that appear in the final itinerary (83–100% across configs)
  • Activity coverage — fraction of all itinerary stops that mention at least one requested activity
  • Enrichment quality — % stops with a real rating, % with review snippets, % with photos (from the rate_pois ToolMessage)
  • LLM-as-judge match quality — an LLM rates each activity-matched stop 1–5 on how well it suits the requested activity, using stop name + description as context (avg_match_quality); averaged across all Track 1 stops

What we did NOT measure (honest gap for the audience):

  • Geographic proximity — we never measured whether selected POIs are spatially close to the actual OSMnx route; the KnowledgeRAG Stage 2 filter checks on-route bounding box but no haversine distance metric was tracked in eval

Recommended config: Run 5 (Tavily + TripAdvisor) — best recall (97%) at competitive latency (43.8s).

Week 5–7 Roadmap — Multi-Agent Expansion

OrchestratorAgent
├── DayTripAgent     (existing)  → itinerary + POIs
├── RestaurantAgent  (new)       → meal stops near each POI
└── LodgingAgent     (new)       → hotel search + booking links

RestaurantAgent — new tools

  • find_restaurants_near_stop — Yelp Fusion / Google Places within 500m of each stop
  • rank_by_cuisine_match — match user preferences from original NL query
  • check_hours — filter closed spots at the planned visit time

LodgingAgent — new tools

  • search_hotels_in_city — Booking.com / SerpAPI hotel search
  • filter_by_budget_proximity — price range + walking distance to first stop next morning
  • generate_booking_link — deep-link to provider (no live reservation needed to start)

Multi-agent evaluation additions

  • Inter-agent coherence — does the restaurant fit the time slot the DayTripAgent allocated?
  • Constraint propagation — does a $150/night budget cap flow correctly across all agents?
  • Parallelization gain — latency improvement when RestaurantAgent + LodgingAgent run concurrently
